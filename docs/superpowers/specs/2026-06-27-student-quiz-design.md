# student-quiz — дизайн

Дата: 2026-06-27 (обновлено по факту реализации, v1.2)
Статус: реализовано

> Документ синхронизирован с текущей моделью. Историю решений см. в git.

## 1. Что это

Веб-приложение (PWA) для самопроверки студентов через тесты. Студент открывает
его в браузере (или ставит как PWA) с телефона/компа, **выбирает тему из меню**,
проходит её, получает балл с разбором и возвращается в меню. Прогресс нигде не
сохраняется — каждый заход в тему начинается с чистого листа.

Крутится в контейнере (OrbStack), доступно из tailnet по Tailscale.

## 2. Требования

### Функциональные
- Главный экран — **меню тем**: список тестов (название + число вопросов).
- Выбор темы → её вопросы одной лентой. Вопрос = текст + варианты:
  - `single` — один верный (radio);
  - `multi` — несколько верных (checkbox, засчитывается по точному совпадению).
- Порядок вопросов и вариантов **перемешивается** на каждое прохождение.
- В конце темы — **итоговый балл + разбор** по каждому вопросу. Кнопки «До тем»
  и «Пройти заново».
- **Прогресс не сохраняется**: нет «продолжить», нет статусов/оценок в меню,
  перезагрузка возвращает в меню. После прохождения тема снова «пустая».
  Студент сам помнит, что уже прошёл.
- База тестов готовится заранее как JSON-файлы; **смена базы на бэкенде** —
  подмена файлов в смонтированном каталоге.
- **Дебаг-режим** (флаг в manifest): показ уникального id вопроса, экран
  «Усі питання» (все вопросы с верными ответами), кнопка «Скинути» в вопросе.
- Доступ: браузер/PWA, телефон/компьютер. Запуск из контейнера OrbStack,
  наружу через Tailscale.

### Нефункциональные
- Стек как в `flora-puzzle`: Vite + TypeScript (vanilla) + PWA, nginx-контейнер,
  доступ по `http://<tailscale-имя>:port`.
- Язык интерфейса и контента — украинский.

### Вне скоупа (YAGNI)
- Серверная проверка ответов и хранение результатов.
- Идентификация/логин студентов.
- Сохранение прогресса, возобновление, статусы прохождения.
- Админ-UI загрузки базы (смена — подмена файлов в volume).

> Модель угрозы — **не противник**: студент сознательно тренируется, не лезет в
> код и не списывает. Поэтому правильные ответы остаются на клиенте (видны в
> DevTools), а перемешивание служит качеству тренировки, а не защите от взлома.

## 3. Архитектура

Вариант **A**: статичный SPA + nginx + volume с JSON. Серверного кода нет.

```
manifest.json (version, title, debug?, список тестов)
   └─> test-XX.json (id, title, questions[])

Браузер:
  fetch /tests/manifest.json → fetch каждого теста (тесты хранятся раздельно)
  → меню тем → выбор → перемешка вопросов/вариантов (в памяти)
  → ответы в памяти → grader на клиенте → балл + разбор → назад в меню
```

### Компоненты

**`src/core/` — чистый TypeScript, без DOM, покрыт vitest:**
- `loader` — грузит и валидирует manifest + тесты; битый JSON отбраковывается
  с понятным сообщением; проверяет уникальность id вопросов между файлами.
- `grader` — по списку вопросов и ответов считает балл и разбор.
- `shuffle` — Фишер–Йетс + `buildOrder(questions)` (перемешка вопросов и опций
  одного теста).

**`src/ui/`** — `screens.ts` (renderMenu / renderQuestion / renderResult /
renderAllQuestions / renderMessage) + `app.ts` (контроллер: меню → прогон в
памяти → результат). Состояние прохождения — **только в памяти** (`Run`),
localStorage не используется.

**Контейнер:** multi-stage `Dockerfile` (vite build → nginx), `compose.yaml`,
каталог `tests/` смонтирован как volume (`:ro`); nginx отдаёт `/tests/*` с
`Cache-Control: no-store`.

## 4. Модель данных

```jsonc
// tests/manifest.json
{
  "version": "v1.2",      // косметический ярлык версии базы
  "title": "Підготовка КОРД",
  "debug": false,          // true → дебаг-режим (id, «Усі питання», «Скинути»)
  "tests": ["test-01.json", "test-02.json", ...]   // порядок в меню
}

// tests/test-XX.json
{
  "id": "test-01",
  "title": "Тема",
  "questions": [
    { "id": "q1", "type": "single",            // radio — ровно 1 верный
      "text": "…", "options": [{ "id": "a", "text": "…" }, …], "correct": ["a"] },
    { "id": "q2", "type": "multi",             // checkbox — 1+ верных
      "text": "…", "options": [ … ], "correct": ["a", "c"] }
  ]
}
```

- В коде база грузится в `LoadedQuiz { version, title, debug?, tests: Test[] }` —
  тесты **хранятся раздельно** (для меню), не склеиваются в одну ленту.
- Прохождение — `Run { test, order, answers, index }` в памяти; нигде не
  персистится.
- Правила оценивания: `single` — выбран ровно верный вариант; `multi` — точное
  совпадение множества (всё-или-ничего). Балл = верных / всего.
- `id` вопроса уникален в пределах всей базы (loader отбраковывает дубликаты).

## 5. UX-поток

```
[Меню тем]  список тестов (назва · N питань)
   │ tap теми
   ▼
[Питання]  N / M (в рамках теми), прогрес-бар
  • Назад / Далі одразу під лічильником (фіксована позиція)
  • radio (single) / checkbox (multi); останнє → «Завершити»
   ▼
[Результат теми]  балл + разбор → «До тем» (меню) / «Пройти заново»
```

Перезагрузка/закрытие — возврат в меню (ничего не сохранено). После «До тем»
тема снова доступна с чистого листа.

**Обработка ошибок:** битый/неполный тест → отбраковка с сообщением; дубликаты
id → ошибка; пустая база → «Тестів поки немає»; офлайн (PWA) → app-shell и тесты
из кэша последней загрузки.

## 6. Контейнер и доступ

- `docker compose up -d --build` в OrbStack.
- Доступ из tailnet: `http://<tailscale-имя>:8089` с телефона/компа.
- PWA-установка требует HTTPS → `tailscale serve --bg 8089`.
- Смена базы: заменить/добавить JSON в `tests/` на хосте (рестарт не нужен,
  `no-store`). Изменения **данных** подхватываются live через volume; изменения
  **кода** требуют `--build`.

## 7. Перемешивание (качество тренировки)

Перемешиваются порядок вопросов и порядок вариантов внутри каждого вопроса
(Фишер–Йетс), заново на каждое прохождение. На оценку не влияет (сравнение по
множествам id). Это средство тренировки (не заучивать позицию), а не защита от
взлома — см. модель угрозы в разделе 2.

## 8. Дебаг-режим

Флаг `"debug"` в `manifest.json` (переключается подменой файла на сервере):
- у вопроса рядом со счётчиком — уникальный `id` (удобно сообщать, какой вопрос
  править);
- в меню — кнопка «Усі питання»: read-only список всех вопросов всех тем с
  отмеченными верными ответами;
- в вопросе — кнопка «Скинути»: бросить прохождение и вернуться в меню.

В репозитории дефолт `debug: false`; включается оперативно на сервере.

## 9. Тестирование

- **Юнит (vitest)** для `core`: `loader` (валидация, дубликаты id),
  `grader` (single/multi, всё-или-ничего), `shuffle` (перестановка, детерминизм),
  `screens` (меню/«Усі питання» рендер).
- **Приёмочное ручное тестирование пользователем** после каждой фазы — гейт:
  разработчик доводит до «запущено в контейнере, доступно по Tailscale», далее
  пользователь сам прокликивает с телефона и компа и даёт «ок».

## 10. Содержание базы (v1.2)

12 тем / 115 вопросов. Источники (выверенные):
- `Картка_Поліція_мобільна` — поліція/Інструкція, ПМ ТТХ, гранати;
- шпаргалка `Співбесіда УПО` (сверена с zakon.rada) — правові основи,
  спецзасоби (ст. 45), застосування зброї (ст. 46), заходи безпеки (Наказ 70),
  **дослівні визначення** (сложный тест: верный = полное определение,
  дистракторы — те же с точечными искажениями);
- docx ТТХ — Форт-12/17; плакат — АКС-74У;
- `Стрельбы Лекции` — основи стрільби (#3), затримки при стрільбі (#2).

Гранатомёты (РПГ-7, лекция #6) — намеренно вне базы. Контент дорабатывается
итеративно.

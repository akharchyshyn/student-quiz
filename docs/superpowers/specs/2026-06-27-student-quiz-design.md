# student-quiz — дизайн

Дата: 2026-06-27
Статус: утверждён (брейншторм)

## 1. Что это

Веб-приложение для самопроверки студентов через тесты. Студент открывает
приложение в браузере (или ставит как PWA) с телефона или компа, проходит
тесты одной непрерывной лентой и получает итоговый балл с разбором.

Приложение крутится в контейнере (OrbStack) и доступно из tailnet по Tailscale.

## 2. Требования

### Функциональные
- Главный экран с кнопкой **«Старт»** (нет прогресса) или **«Продолжить»**
  (есть сохранённый прогресс), плюс «Начать заново».
- «Старт»/«Продолжить» ведёт в ленту вопросов.
- Вопрос = текст + варианты ответа. Тип вопроса:
  - `single` — один верный ответ (radio);
  - `multi` — несколько верных (checkbox).
- База тестов (вопросы + правильные ответы) готовится заранее как JSON-файлы.
- Несколько активных тестов склеиваются в **одну непрерывную ленту** вопросов;
  в конце — **один итоговый балл + разбор** по каждому вопросу.
- Прогресс (текущий вопрос + ответы) сохраняется **на стороне клиента**
  (localStorage). Закрытие вкладки/телефона не теряет прогресс.
- На бэкенде есть возможность **сменить базу тестов** без пересборки образа.
- Доступ: браузер или PWA, телефон или компьютер.
- Запуск из контейнера OrbStack, доступ наружу через Tailscale-сеть.

### Нефункциональные
- Паттерн и стек — как в соседнем проекте `flora-puzzle`
  (Vite + TypeScript + PWA, nginx-контейнер, доступ по `http://<tailscale-имя>:port`).
- Тексты UI и документация — на русском.

### Явно вне скоупа (YAGNI)
- Серверный сбор/хранение результатов.
- Идентификация/логин студентов.
- Админ-UI для загрузки базы (смена — подмена файлов в volume).
- Серверная проверка ответов и скрытие правильных ответов от клиента.

> Следствие принятых решений: правильные ответы лежат в JSON, который отдаётся
> клиенту, поэтому технически их видно в DevTools/сети. Для режима самопроверки
> это допустимо и принято осознанно.

## 3. Архитектура

Вариант **A**: статичный SPA + nginx + volume с JSON. Серверного кода нет.

```
manifest.json (порядок активных тестов)
   └─> test-01.json, test-02.json, … (вопросы + правильные ответы)

Браузер:
  fetch /tests/manifest.json
    → fetch каждого файла из tests[]
    → склейка всех вопросов в одну ленту
    → ответы пишутся в localStorage по мере прохождения
    → grader (на клиенте) считает балл
    → экран результата + разбор
```

### Компоненты

**`core/` — чистый TypeScript, без DOM, покрыт юнит-тестами (vitest):**
- `loader` — грузит и **валидирует** manifest и тесты; битый/неполный JSON
  отбраковывается с понятным сообщением, приложение не падает.
- `grader` — по набору ответов считает балл и формирует разбор
  (верно/неверно по каждому вопросу, что было выбрано, что правильно).
- `progress` — чтение/запись прогресса в localStorage + контроль версии базы.

**`ui/` — слой представления:**
- Экран «Главная» (Старт/Продолжить/Начать заново).
- Экран «Вопрос» (radio/checkbox, навигация, прогресс-бар).
- Экран «Результат» (балл + разбор).

**Контейнер:**
- multi-stage `Dockerfile`: стадия сборки (node + `vite build`) → nginx,
  раздающий собранную статику.
- `compose.yaml`: проброс порта (напр. `8089:80`), монтирование `./tests`
  как volume в read-only.
- nginx-конфиг: `/tests/*` отдаётся с `Cache-Control: no-store`, чтобы новая
  база подхватывалась без рестарта и без кэш-залипания.

## 4. Модель данных

```jsonc
// tests/manifest.json
{
  "version": "2026-06-27a",        // меняется при смене базы; ключ инвалидации прогресса
  "title": "Тесты",
  "tests": ["test-01.json"]        // порядок = порядок ленты
}

// tests/test-01.json
{
  "id": "test-01",
  "title": "Тема 1",
  "questions": [
    {
      "id": "q1",
      "type": "single",            // radio — ровно один верный
      "text": "Текст вопроса?",
      "options": [
        { "id": "a", "text": "Вариант A" },
        { "id": "b", "text": "Вариант B" }
      ],
      "correct": ["a"]
    },
    {
      "id": "q2",
      "type": "multi",             // checkbox — один и более верных
      "text": "Выберите все верные",
      "options": [
        { "id": "a", "text": "…" },
        { "id": "b", "text": "…" },
        { "id": "c", "text": "…" }
      ],
      "correct": ["a", "c"]
    }
  ]
}
```

### Прогресс (localStorage)
```jsonc
{
  "version": "2026-06-27a",        // из manifest на момент старта
  "answers": { "q1": ["a"], "q2": ["a"] },
  "index": 1,                      // текущий вопрос в ленте
  "startedAt": "…",
  "finishedAt": null
}
```

### Правила
- `single`: засчитывается, если выбран ровно верный вариант.
- `multi`: засчитывается по **точному совпадению множества** (всё-или-ничего);
  частично верный набор = неверно.
- Балл = число верных вопросов / общее число вопросов.
- **Идентификация вопросов**: `id` уникален в пределах ленты. Если в разных
  файлах встречаются одинаковые `id`, loader отбраковывает базу с ошибкой
  (во избежание коллизий ключей прогресса).
- **Смена версии базы**: если `version` в manifest отличается от сохранённой в
  прогрессе — показываем «база обновилась, начать заново», старый прогресс
  очищается (вопросы могли измениться, мэппинг ответов недостоверен).

## 5. UX-поток

```
[Главная]
  ├─ нет прогресса → «Старт»
  └─ есть прогресс  → «Продолжить» (+ «Начать заново»)
        ↓
[Вопрос]  (N из M, прогресс-бар)
  • radio (single) / чекбоксы (multi)
  • «Назад» / «Далее»; ответ сохраняется в localStorage сразу
  • последний вопрос → «Завершить»
        ↓
[Результат]  балл (напр. 8/10) + разбор по каждому вопросу
  └─ «Пройти заново» (очистка прогресса → Главная/Старт)
```

### Обработка ошибок и крайние случаи
- Битый/неполный тест → loader отбраковывает с понятным сообщением.
- Дубликаты `id` вопросов между файлами → ошибка валидации базы.
- Пустой manifest / нет тестов → экран «тестов пока нет».
- Сетевая ошибка загрузки тестов (офлайн) → app-shell из кэша PWA, тесты из
  кэша последней успешной загрузки; если кэша нет — понятное сообщение.

## 6. Контейнер и доступ

- `docker compose up -d --build` в OrbStack.
- Доступ из tailnet: `http://<tailscale-имя-машины>:8089` с телефона/компа.
- PWA-установка на домашний экран требует HTTPS → `tailscale serve`
  (инструкция в README).
- **Смена базы тестов**: положить/заменить JSON-файлы в `tests/` на хосте и
  при необходимости обновить `manifest.json` (`version` + список `tests`).
  Volume read-only для контейнера; правки делаются на хосте; рестарт не нужен.

## 7. Фазы

| Фаза | Содержание | Критерий готовности (Definition of Done) |
|---|---|---|
| **POC** | 1 тест-заглушка: 2–3 вопроса, оба типа (radio + checkbox) | Полный цикл Старт → вопросы → балл → разбор → Продолжить работает в контейнере, доступен по Tailscale с телефона и компа |
| **MVP** | 1–4 реальных теста, прогон | Реальные вопросы проходятся, балл корректен, смена базы через подмену файлов в volume проверена вручную |
| **Финал** | добавить все тесты | Все тесты лежат в `tests/`, `manifest.json` обновлён, прогон проходит |

## 8. Тестирование

- **Юнит (vitest)** для `core`: `loader` (валидация, отбраковка битых данных,
  дубликаты id), `grader` (single/multi, всё-или-ничего, подсчёт балла),
  `progress` (запись/чтение, инвалидация по версии).
- **Ручной прогон** UI в контейнере по Tailscale на каждой фазе (телефон + комп).
